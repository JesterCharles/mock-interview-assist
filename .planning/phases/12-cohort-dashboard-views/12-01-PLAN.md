  <action>
Modify `src/app/api/trainer/route.ts`:

1. Change handler signature: `export async function GET(request: Request)`.
2. Parse query: `const cohortId = new URL(request.url).searchParams.get('cohortId')`.
3. Treat `cohortId === null || cohortId === 'all' || cohortId === ''` as "no filter".
4. Build prisma where clause:
   ```ts
   const where = cohortId && cohortId !== 'all' ? { cohortId } : undefined
   ```
5. Pass `where` to existing `prisma.associate.findMany({ where, include: {...}, orderBy: {...} })`.
6. After mapping to `RosterAssociate[]`, compute summary ONLY when filter active:
   ```ts
   const summary = where
     ? rosterData.reduce(
         (acc, a) => {
           if (a.readinessStatus === 'ready') acc.ready++
           else if (a.readinessStatus === 'improving') acc.improving++
           else acc.notReady++
           return acc
         },
         { ready: 0, improving: 0, notReady: 0 }
       )
     : null
   ```
7. Return `NextResponse.json<RosterResponse>({ associates: rosterData, summary })`.

CRITICAL: Per D-02, associates with `cohortId = null` MUST still appear in the unfiltered "All Associates" view. The `where: undefined` ensures Prisma applies no filter, preserving v1.0 behavior. Do NOT add `where: { cohortId: { not: null } }` — that would break backward compatibility (success criterion 4).

Per D-06: aggregation is server-side, computed from the already-filtered list (single query, no extra DB roundtrip).
  </action>
