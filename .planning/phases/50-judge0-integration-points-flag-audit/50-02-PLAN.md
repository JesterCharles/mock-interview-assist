<objective>
Guard **every server-side Judge0 / coding call site** with `isCodingEnabled()`. When the flag is off, API routes return HTTP 503 with the `coming soon` body; library functions throw `CodingFeatureDisabledError`. Add the unauthenticated `/api/coding/status` probe endpoint that Plan 03's UI reads to decide whether to render the normal workspace or the ComingSoon card.

Purpose: JUDGE-INTEG-02 requires that with `CODING_CHALLENGES_ENABLED=false`, calling any `/api/coding/*` route returns "coming soon" AND gap-score persistence hooks are short-circuited. Missing any one call site = audit fails = T-50-01 (coding code runs server-side without flag check).

- `src/app/api/coding/status/route.ts` (NEW) — public probe endpoint

- 3 lib services (`judge0Client.ts`, `codingSignalService.ts` [documented no-op], `codingAttemptPoll.ts`) extended with guard
- Extended tests for flag-off paths on `submit`, `status`, `judge0Client`

    ```typescript
    /**
     * GET /api/coding/status — Phase 50 (JUDGE-INTEG-02 / D-06)
     *
     * Public feature-availability probe. Used by /coding + /coding/[id] + SubmitBar
     * to decide whether to render the normal workspace or the ComingSoon card.
     *
     * - NO auth: rendering cost of ComingSoon card must not require sign-in
     * - Cache-Control public, s-maxage=60 (D-07): bounded load if traffic spikes
     * - ALWAYS 200: this is a probe, never 503 (callers branch on body.enabled)
     * - Response shape: { enabled: boolean } — nothing else leaked
     */

    import { NextResponse } from 'next/server';
    import { isCodingEnabled } from '@/lib/codingFeatureFlag';

    export async function GET(): Promise<NextResponse> {
      const enabled = isCodingEnabled();
      return NextResponse.json(
        { enabled },
        {
          status: 200,
          headers: { 'Cache-Control': 'public, s-maxage=60' },
        },
      );
    }
    ```
