<objective>
Ship GET /api/coding/attempts/[id] polling endpoint with a strict response-shape contract that enforces the hidden-test shield, plus the `codingAttemptPoll.ts` helper that aggregates Judge0 per-test submissions, computes the canonical verdict + server-computed score, persists idempotently, and fires the CodingSkillSignal writeback without blocking the response.

Purpose: This is the second half of the async-submit pattern. The client polls this endpoint until `verdict !== 'pending'`. Hidden test leak here is a product-critical vulnerability (discovery §Cross-Model Perspective §5 + Phase 37 D-05 drove private-repo storage); the response shape is therefore Zod-validated on the WAY OUT, not just on the way in.

       ```typescript
       import { NextRequest, NextResponse } from 'next/server';
       import { z } from 'zod';
       import { getCallerIdentity } from '@/lib/identity';
       import { prisma } from '@/lib/prisma';
       import { pollAndMaybeResolveAttempt, AttemptNotFoundError } from '@/lib/codingAttemptPoll';

       // Zod output schema — enforced BEFORE serialization. This is the hidden-test shield.
       const AttemptResponseSchema = z.object({
         attemptId: z.string(),
         verdict: z.enum(['pending', 'pass', 'fail', 'timeout', 'mle', 'runtime_error', 'compile_error']),
         score: z.number().min(0).max(100).nullable(),
         visibleTestResults: z.array(z.object({
           caseId: z.string(),
           passed: z.boolean(),
           stdout: z.string().nullable(),
           durationMs: z.number().nullable(),
         })),
         hiddenTestResults: z.object({
           passed: z.number().int().min(0),
           total: z.number().int().min(0),
         }),  // strict — no .passthrough(), no array allowed
         submittedAt: z.string(),     // ISO
         completedAt: z.string().nullable(), // ISO or null
       }).strict();  // reject unknown keys

       export async function GET(request: NextRequest, { params }: { params: Promise<{ id: string }> }) {
         const { id } = await params;

         const caller = await getCallerIdentity();
         if (caller.kind === 'anonymous') {
           return NextResponse.json({ error: { code: 'AUTH_REQUIRED', message: 'Sign-in required' } }, { status: 401 });
         }

         // Load attempt with challenge.skillSlug + associateId
         const attempt = await prisma.codingAttempt.findUnique({
           where: { id },
           select: { id: true, associateId: true, challengeId: true, verdict: true, submittedAt: true },
         });
         if (!attempt) {
           return NextResponse.json({ error: { code: 'NOT_FOUND', message: 'Attempt not found' } }, { status: 404 });
         }

         // Authz
         if (caller.kind === 'associate' && attempt.associateId !== caller.associateId) {
           return NextResponse.json({ error: { code: 'FORBIDDEN', message: 'Not your attempt' } }, { status: 403 });
         }

         // Delegate aggregation + persistence + signal to helper
         let pollResult;
         try {
           pollResult = await pollAndMaybeResolveAttempt(id);
         } catch (err) {
           if (err instanceof AttemptNotFoundError) {
             return NextResponse.json({ error: { code: 'NOT_FOUND', message: 'Attempt not found' } }, { status: 404 });
           }
           console.error('[coding/attempts] poll failed:', err);
           return NextResponse.json({ error: { code: 'INTERNAL', message: 'Poll failed' } }, { status: 500 });
         }

         // Build response, validate via Zod
         const attemptFresh = await prisma.codingAttempt.findUnique({
           where: { id },
           select: { submittedAt: true, completedAt: true },
         });

         const responseBody = {
           attemptId: id,
           verdict: pollResult.verdict,
           score: pollResult.score,
           visibleTestResults: pollResult.visibleTestResults,
           hiddenTestResults: pollResult.hiddenAggregate,
           submittedAt: attemptFresh!.submittedAt.toISOString(),
           completedAt: attemptFresh?.completedAt?.toISOString() ?? null,
         };

         const parsed = AttemptResponseSchema.safeParse(responseBody);
         if (!parsed.success) {
           console.error('[coding/attempts] SHIELD TRIPPED — response schema violation:', parsed.error);
           return NextResponse.json({ error: { code: 'INTERNAL', message: 'Response shape error' } }, { status: 500 });
         }

         return NextResponse.json(parsed.data, { status: 200 });
       }
       ```

       Note: The Zod `.strict()` + explicit `hiddenTestResults` object schema means ANY future regression that accidentally surfaces hidden case details will cause the schema parse to fail → 500 INTERNAL. Never ships leak.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-39-09 | Information Disclosure | Hidden test leak via poll response (DevTools inspect) | mitigate | Zod `.strict()` output schema explicitly defines `hiddenTestResults: { passed, total }` — any regression serializing detail triggers shield (500 INTERNAL, log ops alert). Unit test asserts no hidden fixture strings in response body across all scenarios. |
| T-39-10 | Elevation of Privilege | Associate A polls associate B's attempt by guessing/enumerating id | mitigate | Authz check: associate.associateId MUST match attempt.associateId. cuid ids also make enumeration impractical. |
| T-39-11 | Tampering | Client-supplied score via body | mitigate | GET endpoint has no body. Score is server-computed in `computeScore` from per-case weights; clamped [0,100]. |
| T-39-12 | Denial of Service | Poll flood triggers Judge0 stampede | accept / rate-limit | Polling scope is NOT rate-limited in v1.4 (poll itself is cheap once attempt resolves — short-circuits on terminal verdict). If abuse surfaces, add per-attempt poll rate limit in Phase 44. Mitigation: terminal verdict short-circuit means judge0Client.getSubmission is NOT called for resolved attempts. |
| T-39-13 | Tampering | Signal writeback double-fire creating duplicate GapScore inputs | mitigate | `CodingSkillSignal.attemptId @unique` (Phase 36 D-11). `upsert where: { attemptId }` guarantees single row per attempt. |
| T-39-14 | Availability | Signal writeback failure losing gap data silently | mitigate | Wrapped in try/catch + console.error (non-fatal for poll response). Phase 41 ops dashboard surfaces signal-writeback error count. For v1.4 missing signal = missing gap data point, not user-visible break. |
| T-39-15 | Tampering | Race: two concurrent polls both attempt resolution | mitigate | Prisma update uses `where: { id, verdict: 'pending' }` — if already terminal, update affects 0 rows → P2025 error caught, treated as "another worker won the race". Signal upsert by unique attemptId is safe either way. |
