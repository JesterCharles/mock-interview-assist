<objective>
Secure the password change flow to require identity verification before allowing password updates.

Purpose: Per D-14, updating/changing password requires either old password verification OR email verification code. The current Security tab calls `supabase.auth.updateUser({ password })` directly with no prior check — a security gap for magic-link-only users. This plan adds the verification step.
Output: Updated Security tab in ProfileTabs with two-path re-verification (old password for password users, email OTP for magic-link users).
</objective>

**Step 2: Detect password-set status on Security tab mount:**
When `activeTab === 'security'`, check `user_metadata.password_set` OR call `getUser()` to check if `Profile.passwordSetAt` is set. Use the `/api/profile` GET response (which includes profile data with `passwordSetAt`). Alternatively, simpler approach: call `supabase.auth.getUser()` and check `user.user_metadata.password_set`. Set `hasPasswordSet` accordingly.

Add a `useEffect` that fires when `activeTab === 'security'`:
```typescript
useEffect(() => {
  if (activeTab !== 'security') return;
  const supabase = createSupabaseBrowserClient();
  supabase.auth.getUser().then(({ data }) => {
    const pwSet = data.user?.user_metadata?.password_set === true;
    setHasPasswordSet(pwSet);
  });
}, [activeTab]);
```

**Step 3: Two-path verification UX:**

**Path A — User has password set (`hasPasswordSet === true`):**
Show "Current password" input field FIRST. On submit:
1. Call `supabase.auth.signInWithPassword({ email: currentUserEmail, password: oldPassword })`.
2. If error: show "Current password is incorrect." error message. Do NOT proceed.
3. If success: set `verificationStep = 'update'` to reveal new password fields.

The email comes from the `email` prop already passed to ProfileTabs.

**Path B — User has NO password (`hasPasswordSet === false`, magic-link only):**
Show "To set a password, we need to verify your identity" message + "Send verification email" button.
1. On button click: call `supabase.auth.reauthenticate()`. Set `otpSent = true`.
2. Show 6-digit OTP input field.
3. On OTP submit: call `supabase.auth.verifyOtp({ email, token: otpCode, type: 'email' })`.
4. If error: show error. If success: set `verificationStep = 'update'`.

**Step 4: Update form flow:**
When `verificationStep === 'update'`:
- Show new password + confirm password fields (existing UI).
- On submit: call `supabase.auth.updateUser({ password: newPassword })`.
- On success: reset all state, show success message, update `passwordSetAt` via `/api/profile` PUT (existing logic).
- Set `verificationStep` back to `'verify'` after success (so next password change requires re-verification).

**Step 5: Replace existing handlePasswordSubmit:**
Remove the current direct `updateUser` call. Replace with the two-step flow. The form structure becomes conditional on `verificationStep`:
- `verify`: show Path A or Path B UI
- `update`: show new password + confirm + submit

**Step 6: Error handling for rate limits (Pitfall 6 from RESEARCH):**
Show clear error on first `signInWithPassword` failure. Display: "Current password is incorrect." Do NOT retry automatically. If rate-limited by Supabase, show the error message from Supabase verbatim.

**Styling:** Use the existing `inputBase`, `labelStyle`, `fieldGroup` styles already in ProfileTabs. OTP input: `textAlign: center`, `letterSpacing: 8px`, `fontSize: 20`, `maxWidth: 200px`. Style the "Send verification email" button with `btn-accent-flat` class.

**Loading state:** While `hasPasswordSet === null`, show a small loading state ("Checking...") in the Security tab.
