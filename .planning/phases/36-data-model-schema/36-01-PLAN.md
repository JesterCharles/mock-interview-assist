```prisma
// ─────────────────────────────────────────────────────────────────────
// v1.4 Coding Challenges (Phase 36)
// ─────────────────────────────────────────────────────────────────────

model CodingChallenge {
  id          String   @id @default(cuid())
  slug        String   @unique
  title       String
  language    String   // 'python' | 'javascript' | 'typescript' | 'java' | 'sql' | 'csharp' (per D-03, Zod-validated at write)
  difficulty  String   // 'easy' | 'medium' | 'hard' (Zod-validated at write)
  description String   // markdown path in public repo (e.g. "challenges/two-sum/README.md")
  skillSlug   String   // joins to CurriculumWeek.skillSlug
  cohortId    Int?
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt

  cohort    Cohort?          @relation(fields: [cohortId], references: [id], onDelete: SetNull)
  attempts  CodingAttempt[]
  testCases CodingTestCase[]

  @@index([cohortId])
  @@index([skillSlug])
  @@index([language])
}

model CodingAttempt {
  id                 String    @id @default(cuid())
  associateId        Int
  challengeId        String
  submittedCode      String    // raw user code, could be large
  language           String    // same allowlist as challenge
  verdict            String    @default("pending")
                               // 'pass' | 'fail' | 'timeout' | 'mle' | 'runtime_error' | 'compile_error' | 'pending'
  visibleTestResults Json      @default("[]")
                               // array of {caseId, passed, stdin?, stdout?, expected?, durationMs}
  hiddenTestResults  Json      @default("[]")
                               // array of verdict-only {caseId, passed, durationMs} — NEVER stdin/expected (D-06)
  score              Float?    // 0-100, server-computed, null until Judge0 callback resolves (D-07)
  judge0Token        String?   // nullable — filled in by Phase 39 when Judge0 returns
  submittedAt        DateTime  @default(now())
  completedAt        DateTime?

  associate Associate          @relation(fields: [associateId], references: [id], onDelete: Cascade)
  challenge CodingChallenge    @relation(fields: [challengeId], references: [id], onDelete: Restrict)
  signal    CodingSkillSignal?

  @@index([associateId])
  @@index([challengeId])
  @@index([verdict])
  @@index([submittedAt])
}

model CodingTestCase {
  id             String  @id @default(cuid())
  challengeId    String
  isHidden       Boolean @default(false)
  stdin          String
  expectedStdout String
  weight         Float   @default(1.0) // partial-credit multiplier
  orderIndex     Int     @default(0)

  challenge CodingChallenge @relation(fields: [challengeId], references: [id], onDelete: Cascade)

  @@index([challengeId])
  @@index([isHidden])
}

model CodingSkillSignal {
  id          String @id @default(cuid())
  attemptId   String @unique
  skillSlug   String
  signalType  String // 'pass' | 'partial' | 'fail' | 'compile_error' | 'timeout' (per D-03)
  weight      Float  // 1.0 / 0.85 / 1.0 / 0.6 / 0.8 (per D-16 weight table)
  mappedScore Float  // 0-100

  attempt CodingAttempt @relation(fields: [attemptId], references: [id], onDelete: Cascade)

  @@index([skillSlug])
}
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Schema → runtime | Prisma schema defines storage contract; runtime validation (Zod) enforced in service layer (Phase 39), NOT in DB |
| Associate deletion → coding data | Cascade boundary: deleting Associate scrubs all CodingAttempts (user-scoped PII hygiene) |
| Challenge deletion → attempt integrity | Restrict boundary: attempts reference challenges — cannot delete active challenges (archive pattern for future) |
| Signal write path → GapScore | `@unique` on attemptId prevents retry-duplicate signals inflating readiness math |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-36-01 | Information Disclosure | `CodingTestCase.isHidden` column | mitigate | Schema flags hidden rows via `isHidden Boolean` (D-04); API-layer filter in Phase 39 excludes `where: { isHidden: true }` from client responses. Schema shape supports this — `@@index([isHidden])` enables efficient filtering. |
| T-36-03 | Tampering | `CodingSkillSignal.attemptId` | mitigate | `@unique` constraint (D-11) — DB rejects duplicate signal writes from retry paths. Prevents GapScore inflation from concurrent Judge0 callbacks. |
| T-36-04 | Tampering / data integrity | Orphan signals after attempt/associate deletion | mitigate | `onDelete: Cascade` chain: Associate→Attempt→Signal. Deleting user scrubs all derived signals. Challenge→TestCase also cascades (D-10). |
| T-36-05 | Tampering | `CodingAttempt.score` | mitigate | Schema types score as `Float?` (nullable, server-computed per D-07). Client cannot supply score in API writes — nullable default means "not yet computed by server". Phase 39 enforces no-client-writes contract. |
| T-36-06 | Data integrity | Hidden test result payload shape | accept | `hiddenTestResults Json` is permissive at DB level per D-06 (verdict-only `{caseId, passed, durationMs}` — no stdin/expected). Enforcement is service-layer Zod in Phase 39; schema comment documents contract. Accepted because Prisma Json type is inherently untyped and Zod-at-boundary is project convention. |

## RLS Deferral (per RESEARCH §Security Domain)

Row-Level Security policies for `CodingChallenge`, `CodingAttempt`, `CodingTestCase`, `CodingSkillSignal` are **deferred to Phase 39** (Execution API). Rationale:
- RLS policies mirror application access-control model (associate sees own attempts, trainer sees all)
- Access model is not finalized until the Phase 39 API ships
- Adding RLS in Phase 36 risks writing policies that don't match eventual API shape
- Existing Phase 20 RLS architecture is unchanged; new tables will join it in Phase 39

This deferral does NOT weaken security for Phase 36 because no API routes read/write these tables yet.
</threat_model>
