```typescript
// Phase 33 / SIGNIN-02: Trainer first-login password gate.
// After a successful Supabase login, check user_metadata.password_set before redirecting.
// If the flag is falsy, send the trainer to /auth/set-password instead of /trainer.
// Uses a fresh browser Supabase client because useAuth().user is populated async via
// onAuthStateChange and may not have fired yet by the time this handler resumes.
// Fail-open: if getUser() errors, fall back to nextPath ?? '/trainer' — the middleware
// already blocks unauthenticated access to /trainer, so this cannot produce a bypass (D-07).
async function handleTrainerSubmit(e: FormEvent<HTMLFormElement>) {
  e.preventDefault();
  if (trainerSubmitting) return;
  setTrainerError(null);
  setTrainerSubmitting(true);
  const ok = await login(email, password);
  if (ok) {
    try {
      const supabase = createSupabaseBrowserClient();
      const { data, error } = await supabase.auth.getUser();
      if (error || !data?.user) {
        // Fail-open per D-07 — middleware still blocks unauthenticated access.
        router.replace(nextPath ?? '/trainer');
        router.refresh();
        return;
      }
      const passwordSet = data.user.user_metadata?.password_set === true;
      if (!passwordSet) {
        router.replace('/auth/set-password');
        router.refresh();
        return;
      }
      router.replace(nextPath ?? '/trainer');
      router.refresh();
      return;
    } catch {
      // Fail-open per D-07.
      router.replace(nextPath ?? '/trainer');
      router.refresh();
      return;
    }
  }
  setTrainerError('Invalid email or password.');
  setPassword('');
  setTrainerSubmitting(false);
}
```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Unauthenticated user → /signin form | Untrusted credentials submitted to Supabase auth (existing surface) |
| Unauthenticated user → /api/auth/exchange (magic-link callback) | Tokens or code from email link exchanged for session cookies (existing surface) |
| Authenticated trainer (password not set) → /trainer | **Gate site** — SIGNIN-02 requires redirect to /auth/set-password before /trainer access |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-33-01 | Elevation of Privilege | `src/app/api/auth/exchange/route.ts` role branch | mitigate | Move `passwordSet` check ABOVE `if (role === 'trainer' \|\| role === 'admin')` so trainers cannot bypass the first-login gate via magic-link. Tests T3-A/T3-B in Task 3 verify the gate fires for trainers with passwordSet == null. |
| T-33-02 | Elevation of Privilege | `src/app/signin/SignInTabs.tsx` handleTrainerSubmit | mitigate | Insert `supabase.auth.getUser()` + `user_metadata.password_set` check between successful `login()` and `router.replace`. If flag is falsy, redirect to `/auth/set-password` instead of `/trainer`. Task 4 tests T4-2/T4-3 verify both `undefined` and `false` metadata cases hit the gate. |
| T-33-03 | Denial of Service | Post-login `getUser()` client call | accept | If `getUser()` fails (network/Supabase outage), the handler falls open to `nextPath ?? '/trainer'`. The middleware still blocks unauthenticated access to /trainer. Worst case: broken getUser + working cookie = trainer lands on /trainer without the gate running once. Accepting this risk because (a) fail-closed risks locking out legitimate trainers during transient outages, (b) next navigation will re-run the middleware/route logic, (c) the magic-link path is the harder-to-bypass enforcement point. Documented as D-07 in CONTEXT.md. |
| T-33-04 | Information Disclosure | Client-side `user_metadata.password_set` read | accept | `user_metadata` is visible in the authenticated user object — no new information disclosure. The flag indicates "has user set a password" which is not sensitive (any observer who can read the user object is already authenticated as that user). |
| T-33-05 | Tampering | Client-side metadata-only check (D-06) | accept | A malicious trainer could theoretically tamper with their own `user_metadata.password_set` in the Supabase token, but (a) Supabase signs JWTs with the project secret — tampering breaks the signature, (b) even if bypassed client-side, the exchange route still runs the Profile-first check on magic-link sign-in, (c) the trainer can only "escape" into /trainer for their own session, which they could already do if they set a real password. Not a meaningful attack surface. Full Profile-first detection on the password path is deferred as a `/api/auth/password-status` endpoint (deferred idea, Phase 28.1). |
| T-33-06 | Repudiation | Trainer redirected to gate, then sets password | accept | Existing Supabase audit log + Phase 28.1 lazy-backfill already log password-set events. No new logging needed for this phase. |
| T-33-07 | Spoofing | Associate flow impersonation via reorder | mitigate | The reorder moves the passwordSet check above the role branch — associates already ran it AFTER the (never-hit) trainer branch, so the change is order-equivalent for them. Existing associate tests in `route.test.ts` (lines 87-93, 95-101, 142-174) provide regression coverage. If the reorder accidentally alters associate behavior, those tests fail loudly in Task 1. |
