**1. Create `/auth/set-password` page (per D-09):**

New file: `src/app/auth/set-password/page.tsx` â€” client component.

Pattern: Reference existing `/auth/update-password/page.tsx` for form structure (password + confirm, 8 char min, same `inputBase` styles, same card layout).

Key differences from update-password:
- Heading: "Set your password" (not "Set new password")
- Subheading: "Create a password so you can sign in directly next time."
- Per D-08: No skip/cancel button. No back link. No navigation away. The page is a dead end until password is set.
- After successful `supabase.auth.updateUser({ password })`:
  - Also call `supabase.auth.updateUser({ data: { password_set: true } })` to set the metadata flag (per D-11). These can be combined into one call: `supabase.auth.updateUser({ password, data: { password_set: true } })`.
  - Per D-10: Redirect to role-appropriate dashboard. Use `supabase.auth.getUser()` to read `user_metadata.role`:
    - `role === 'trainer' || role === 'admin'` -> redirect to `/trainer`
    - Otherwise -> need associate slug. Fetch from `/api/associate/me` to get slug, then redirect to `/associate/{slug}/dashboard`. Fallback to `/` if fetch fails.
  - Show brief success message ("Password set. Redirecting...") during the redirect delay (1.5s).

**2. Add first-login detection in `/api/auth/exchange/route.ts` (per D-06):**

After session exchange succeeds and before the role-based redirect logic, insert a `password_set` check:

```typescript
// After getting user, before role-based redirect:
const passwordSet = user.user_metadata?.password_set === true;

// For recovery flow, still go to update-password (existing behavior)
if (type === 'recovery') {
  return redirectWith('/auth/update-password');
}

// First-login detection: if no password set, redirect to set-password (per D-05, D-06, D-07)
if (!passwordSet) {
  return redirectWith('/auth/set-password');
}
```

This goes after the `type === 'recovery'` check but before the trainer/associate role redirect. This applies to BOTH trainers and associates who arrive via magic link (per D-07). Edge case per Claude's Discretion: users who already have a password but no metadata flag will be prompted once, which sets the flag.

Important: Do NOT add this check to the password sign-in flow (that's in `handleTrainerSubmit` in SignInTabs). Password sign-in means user already has a password. Only magic-link/token exchange flows hit `/api/auth/exchange`.

**3. Update middleware to allow `/auth/set-password` (per D-09):**

In `src/middleware.ts`, the `/auth/set-password` route is under `/auth/*` which is NOT in the matcher config (`/dashboard/:path*`, `/interview/:path*`, etc.). So it should already pass through without middleware blocking. Verify this is the case.

However, if the user navigates to `/auth/set-password` directly without being authenticated, the page should handle it gracefully. Add a check in the set-password page: if `supabase.auth.getUser()` returns no user, redirect to `/signin`.

- Add password sign-in for associates on the /signin page (deferred per CONTEXT.md)
- Modify the trainer password sign-in flow (it bypasses /api/auth/exchange entirely)
