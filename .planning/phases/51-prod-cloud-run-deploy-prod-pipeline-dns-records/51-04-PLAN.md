### ## Section 6: Zero-Downtime Invariant

**Invariant:** Any active public-interview session started on v0.1 BEFORE DNS propagation completes on the client MUST complete successfully on v0.1.

**Why it holds:**
- Cloudflare caches pre-cutover apex record per-client up to 300s (post TTL lowering).
- Browsers cache DNS for OS/chrome default (typically 60-300s).
- Sessions in progress keep established TCP connections to v0.1; DNS flip does not abort live connections.
- Sessions are short (10-30 min average); in-flight sessions drain naturally.

**New sessions** started AFTER propagation go to prod Cloud Run. v0.1 and prod share the same Supabase prod DB (wiped + migrated in Phase 46) so no split-brain.

**Post-cutover manual verification (trainer):** Start one session on apex BEFORE expected propagation, observe completion; then a second session AFTER propagation, observe completion. Both should succeed.
