## What Was Built

**Profile model (D-01 through D-04):** New `Profile` table with `authUserId` (unique, Supabase auth.users.id), `displayName`, `githubUsername`, `bio`, `learningGoals`, `passwordSetAt`. Not linked to `Associate` ‚Äî trainers and admins get profiles too. Migration SQL created at `prisma/migrations/20260416000000_add_profile/migration.sql`. `prisma generate` run to update client types.

**profileService.ts:** Three exports ‚Äî `getOrCreateProfile` (upsert, creates on first access), `updateProfile` (partial upsert), `lazyBackfillProfile` (migrates user_metadata ‚Üí Profile.passwordSetAt on login if not yet done).

**/api/profile route (D-03, D-04, T-28.1-01 through T-28.1-03):** GET returns profile + role + email + readiness data for associates (own record only). PUT validates with Zod (field lengths), updates only safe fields. Anonymous ‚Üí 401.

**Profile page /profile (D-05 through D-09):** Server-rendered page with auth guard. Fetches profile and readiness data server-side. Passes to `ProfileTabs` client component. Three tabs:
- **Profile:** Email (read-only), role badge (read-only), editable display name / GitHub username / bio. Saves via PUT /api/profile.
- **Security:** Inline password change form (8 char min, confirm field). Calls `supabase.auth.updateUser`. Also writes `passwordSetAt` to Profile on success.
- **Learning:** Readiness summary card (status badge, score %, focus area, session count) for associates. Editable learning goals textarea. Trainers/admins see placeholder message.

**AvatarMenu (D-10, D-11):** Profile link inserted between role badge and Sign Out: email ‚Üí role badge ‚Üí separator ‚Üí Profile ‚Üí separator ‚Üí Sign Out.

**Middleware:** `/profile/:path*` added to matcher. STEP 4 guard ‚Äî any authenticated user passes, unauthenticated ‚Üí `/signin?next=`.

**First-login detection migration (D-12 through D-14):** Auth exchange route now runs `lazyBackfillProfile` on every login (creates Profile from metadata if needed), then checks `Profile.passwordSetAt` first, falls back to `user_metadata.password_set`. `/auth/set-password` writes `passwordSetAt` to Profile after successful password set (non-blocking try/catch; metadata write preserved for backward compat).
