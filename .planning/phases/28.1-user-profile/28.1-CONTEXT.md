# Phase 28.1: User Profile - Context

**Gathered:** 2026-04-16
**Status:** Ready for planning

<domain>
## Phase Boundary

Add a shared Profile model (Prisma) linked to Supabase authUserId, build a tabbed profile page at `/profile` accessible from the avatar menu, and migrate first-login password detection from user_metadata to Profile.passwordSetAt. Profile serves both trainers and associates. Design for future expandability.

</domain>

<decisions>
## Implementation Decisions

### Profile Model
- **D-01:** Shared `Profile` table for trainers AND associates. Linked via `authUserId` (unique FK to Supabase `auth.users.id`). One profile per Supabase user.
- **D-02:** MVP fields: `passwordSetAt` (DateTime?), `githubUsername` (String?), `displayName` (String?), `bio` (String?), `learningGoals` (String?).
- **D-03:** `authUserId` is the primary link — not tied to Associate model directly. Trainers (who have no Associate record) can still have a Profile.
- **D-04:** Schema designed for expandability — additional fields (avatar URL, notification prefs, timezone, etc.) can be added in future milestones without restructuring.

### Profile Page Layout
- **D-05:** Tabbed sections — horizontal tabs: Profile | Security | Learning.
- **D-06:** Profile tab: email (read-only), role badge (read-only), display name (editable), github username (editable), bio (editable).
- **D-07:** Security tab: change password form (inline, same pattern as `/auth/update-password`).
- **D-08:** Learning tab: readiness summary card (read-only — status, score %, focus area, session count from existing data) + learning goals (editable textarea).
- **D-09:** Route: `/profile`. Auth-guarded — redirects to `/signin` if anonymous. Works for both roles.

### Avatar Menu Integration
- **D-10:** Add "Profile" link to existing AvatarMenu dropdown between role badge and sign out separator. Keep email display and role badge as-is.
- **D-11:** Menu order: email → role badge → separator → Profile → separator → Sign Out.

### Password Migration
- **D-12:** Lazy backfill — on login, if `user_metadata.password_set` is true and no `Profile.passwordSetAt`, create/update Profile record with `passwordSetAt = now()`.
- **D-13:** Detection logic: Profile-first, metadata fallback. Check `Profile.passwordSetAt` first. If no Profile record exists, fall back to `user_metadata.password_set`. Graceful transition.
- **D-14:** `/auth/set-password` (from Phase 28) writes to BOTH `Profile.passwordSetAt` AND `user_metadata.password_set` for backward compatibility during transition.

### Claude's Discretion
- Tab component implementation (custom tabs vs radix-ui tabs)
- Form save UX (auto-save vs explicit save button per section)
- How Learning tab fetches readiness data (server component vs client fetch)
- Profile creation timing (on first login, on first profile page visit, or on first save)
- Whether `/profile` renders inside the unified shell (TopBar+Sidebar) or standalone

</decisions>

<specifics>
## Specific Ideas

- Profile page should feel expandable — tab structure allows adding more tabs (Notifications, Preferences, etc.) in future milestones without layout changes.
- Learning tab's readiness summary is read-only, pulled from existing Associate + GapScore data. No new data pipeline needed.
- Lazy backfill is the simplest path — no migration scripts, handles the transition organically as users log in.

</specifics>
