2. **Create `src/lib/historyService.ts`** — extract the three file-based functions from `route.ts`. This is needed because the sync-check endpoint (Plan 02) must also call `readHistory`. Extract EXACTLY:

```typescript
import fs from 'fs';
import path from 'path';
import { InterviewSession } from '@/lib/types';

const DATA_DIR = path.join(process.cwd(), 'data');
const HISTORY_FILE = path.join(DATA_DIR, 'interview-history.json');

export function ensureDataDir(): void {
  if (!fs.existsSync(DATA_DIR)) {
    fs.mkdirSync(DATA_DIR, { recursive: true });
  }
}

export function readHistory(): InterviewSession[] {
  ensureDataDir();
  if (!fs.existsSync(HISTORY_FILE)) {
    return [];
  }
  try {
    const content = fs.readFileSync(HISTORY_FILE, 'utf-8');
    return JSON.parse(content);
  } catch (error) {
    console.error('Error reading history file:', error);
    return [];
  }
}

export function writeHistory(history: InterviewSession[]): void {
  ensureDataDir();
  fs.writeFileSync(HISTORY_FILE, JSON.stringify(history, null, 2));
}
```

```typescript
// Dual-write to Supabase (D-01: log-and-continue on failure)
try {
  await prisma.session.upsert({
    where: { id: session.id },
    create: {
      id: session.id,
      candidateName: session.candidateName ?? null,
      interviewerName: session.interviewerName ?? null,
      date: session.date,
      status: session.status,
      questionCount: session.questionCount,
      selectedWeeks: session.selectedWeeks as unknown as Prisma.InputJsonValue,
      overallTechnicalScore: session.overallTechnicalScore ?? null,
      overallSoftSkillScore: session.overallSoftSkillScore ?? null,
      technicalFeedback: session.technicalFeedback ?? null,
      softSkillFeedback: session.softSkillFeedback ?? null,
      questions: session.questions as unknown as Prisma.InputJsonValue,
      starterQuestions: session.starterQuestions as unknown as Prisma.InputJsonValue,
      assessments: session.assessments as unknown as Prisma.InputJsonValue,
    },
    update: {
      candidateName: session.candidateName ?? null,
      interviewerName: session.interviewerName ?? null,
      status: session.status,
      overallTechnicalScore: session.overallTechnicalScore ?? null,
      overallSoftSkillScore: session.overallSoftSkillScore ?? null,
      technicalFeedback: session.technicalFeedback ?? null,
      softSkillFeedback: session.softSkillFeedback ?? null,
      questions: session.questions as unknown as Prisma.InputJsonValue,
      starterQuestions: session.starterQuestions as unknown as Prisma.InputJsonValue,
      assessments: session.assessments as unknown as Prisma.InputJsonValue,
    },
  });
} catch (dbError) {
  console.error('[dual-write] Supabase write failed:', dbError);
}
```

Key points:
- File write happens FIRST (source of truth during migration per D-01)
- DB write is wrapped in inner try/catch — failures are logged but never surface to the client
- Use `as unknown as Prisma.InputJsonValue` for all Json columns (per RESEARCH.md Pitfall 2)
- `upsert` handles both new sessions and re-saves of existing ones
- The `update` block includes `questions`, `starterQuestions`, `assessments` because a review-phase re-save may update these
- `?? null` coalesces undefined to null for optional fields (Prisma requires explicit null, not undefined)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client -> /api/history POST | Session payload from authenticated client crosses into server-side persistence |
| /api/history -> Supabase | DB write crosses network boundary to hosted Postgres |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | /api/history POST body | mitigate | POST handler checks `isAuthenticatedSession()` before processing. Session payload is typed as `InterviewSession` — malformed data will fail at Prisma upsert (type mismatch on Int/Float columns). Full zod validation deferred to a future hardening pass but Prisma schema enforces column types at write time. |
| T-02-02 | Information Disclosure | DB error in dual-write catch | mitigate | `console.error` only — error message is NEVER included in the API response. Response always returns `{ success: true }` regardless of DB write outcome. |
| T-02-03 | Denial of Service | Large JSON payload in assessments column | accept | Existing session payloads are bounded by question count (max ~20 questions). Supabase Postgres has a 1GB row limit — session payloads are KB-scale. No additional size validation needed for MVP. |
| T-02-04 | Elevation of Privilege | Unauthenticated write to Session table | mitigate | `isAuthenticatedSession()` guard on POST handler. Prisma singleton does not expose a public API — only server-side route handlers can call it. |
</threat_model>
