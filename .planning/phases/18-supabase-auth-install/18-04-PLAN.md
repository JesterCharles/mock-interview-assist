       - Rate limit: `checkAuthRateLimit({ email, ip, type: 'magic-link' })`. Return 429 if blocked.
       - Generate link: `supabaseAdmin.auth.admin.generateLink({ type: 'magiclink', email, options: { redirectTo: '${process.env.NEXT_PUBLIC_SITE_URL}/auth/callback' } })`.
       - If error or no action_link: return 200 `{ ok: true }` anyway (do not reveal whether user exists). Log error server-side.
       - Send via Resend: `resend.emails.send({ from: 'Next Level Mock <noreply@nextlevelmock.com>', to: email, subject: 'Sign in to Next Level Mock', html: getMagicLinkEmailHtml(data.properties.action_link) })`.

       - Mock supabaseAdmin, Resend, checkAuthRateLimit, recordAuthEvent.

    3. **Create `src/app/auth/callback/route.ts`**:
       - `export async function GET(request: NextRequest)` — extract `code` and `next` from searchParams.
       - If no code: redirect to `/signin?error=missing-code`.
       - Create Supabase server client, call `exchangeCodeForSession(code)`.
       - If exchange error: redirect to `/signin?error=invalid-link`.
       - On success: get user via `supabase.auth.getUser()`.
       - **authUserId linkage** (Pitfall 5 from RESEARCH.md): If user exists and role is associate (or default), check `prisma.associate.findUnique({ where: { authUserId: user.id } })`. If no match, attempt `prisma.associate.findUnique({ where: { email: user.email } })`. If email match found AND `authUserId` is null, update: `prisma.associate.update({ where: { id: associate.id }, data: { authUserId: user.id } })`. This links the Supabase user to the Associate row on first sign-in.
       - **Race handling** (Codex finding): Wrap the `prisma.associate.update` in a try/catch. If a unique-constraint error occurs on `authUserId` (P2002), another concurrent callback already linked the row — re-read by `authUserId` and continue with the existing linkage. Do NOT treat this as a fatal error.
       - Determine redirect: if `user.user_metadata?.role` is `trainer` or `admin`, redirect to `next ?? '/trainer'`. If associate with linked Associate row, redirect to `/associate/${associate.slug}` (or `next` if provided). If associate with no linkable row, redirect to `/signin?error=not-onboarded` (associate email not in system).
       - For recovery type (`searchParams.get('type') === 'recovery'`): redirect to `/auth/update-password` (implemented in Plan 03 Task 2 step 4).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser -> /api/auth/magic-link | Untrusted email; must not leak user existence |
| /auth/callback -> Supabase | PKCE code exchange; must validate before granting session |
| /auth/callback -> database | authUserId linkage; must only link when email matches |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-18-14 | Spoofing | /api/auth/magic-link | mitigate | PKCE flow — email scanners cannot complete exchange without browser's code verifier cookie |
| T-18-15 | Information Disclosure | /api/auth/magic-link | mitigate | Always return 200 regardless of email existence |
| T-18-16 | Denial of Service | /api/auth/magic-link | mitigate | 3/hr/email + 10/hr/IP rate limiting |
| T-18-17 | Elevation of Privilege | /auth/callback authUserId linkage | mitigate | Only link authUserId when Associate.email matches Supabase user.email AND authUserId is currently null (prevent hijack) |
| T-18-18 | Spoofing | /auth/callback | mitigate | exchangeCodeForSession validates PKCE code + verifier; invalid codes rejected and redirected to /signin?error |
| T-18-19 | Repudiation | Magic link requests | mitigate | All requests logged to AuthEvent table |
</threat_model>
