    1. **Create `src/lib/email/auth-templates.ts`** — export two template functions:

Follow the existing `getReportEmailHtml` pattern in `src/lib/email-templates.ts` for HTML structure and inline styles. Use design tokens from DESIGN.md (accent color, font family). Include a "If you didn't request this, ignore this email" footer.
       - `getMagicLinkEmailHtml(magicLink: string): string` — branded HTML email with "Sign in to Next Level Mock" heading, a button linking to the magic link URL, and a plain-text fallback. Include "This link expires in 7 days" note. Same styling pattern.
    2. **Create `src/app/api/auth/reset/request/route.ts`**:
       - Validate body with Zod: `z.object({ email: z.string().email() })`.
       - Extract IP from `x-forwarded-for` header (respect `NLM_TRUSTED_PROXY` env var) or fall back to `'unknown'`.
       - Call `checkAuthRateLimit({ email, ip, type: 'reset' })`. If not allowed, return 429 with `{ error: 'Too many requests' }`.
       - Call `supabaseAdmin.auth.admin.generateLink({ type: 'recovery', email, options: { redirectTo: '${NEXT_PUBLIC_SITE_URL}/auth/callback' } })`.
       - If generateLink returns error or no action_link, return 200 anyway (do not leak whether email exists — security best practice).
       - If successful, send via Resend: `resend.emails.send({ from: 'Next Level Mock <noreply@nextlevelmock.com>', to: email, subject: 'Reset your Next Level Mock password', html: getResetEmailHtml(actionLink) })`.
       - Record event: `recordAuthEvent({ type: 'reset', email, ip })`.
       - **Abuse flag logic**: After recording the reset event, query `AuthEvent` count for this email in the last 24h where type='reset'. If count >= 5:
         - First CHECK: query if a `reset-abuse-flag` event for this email exists in the last 24h (deduplication). If one already exists, skip.
         - If no existing flag: THEN record `recordAuthEvent({ type: 'reset-abuse-flag', email, ip, metadata: { flagCount: count } })`.
         - After recording the flag: send admin notification email to each address in `ADMIN_EMAILS` env var via Resend: subject "Password reset abuse flag: {email}", body with count and timestamp.
         - ORDERING IS CRITICAL: check-then-record prevents the dedupe query from finding the just-created flag and suppressing the first admin email.
       - Always return `{ ok: true }` (200) regardless of whether email exists. Do not leak user existence.
    3. Import `supabaseAdmin` from `@/lib/supabase/admin`, `Resend` from `resend`, `checkAuthRateLimit` and `recordAuthEvent` from `@/lib/authRateLimit`.
    4. **Create `src/app/auth/update-password/page.tsx`** — password update completion page:
       - Client component (`'use client'`). Import `createSupabaseBrowserClient` from `@/lib/supabase/browser`.
       - On mount: Supabase client auto-detects the recovery token from the URL hash (handled by `@supabase/ssr` after `/auth/callback` redirects here).
       - Form: two fields — "New password" and "Confirm password". Client-side validation: min 8 chars, fields must match.
       - On submit: call `supabase.auth.updateUser({ password: newPassword })`. On success: show "Password updated" confirmation + redirect to `/signin` after 2s. On error: show error message inline.
       - Follow DESIGN.md styling (`inputBase`, card layout, centered form). Minimal page — no nav, just the form.
       - The `/auth/callback` route (Plan 04) redirects `type=recovery` to this page.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser -> /api/auth/reset/request | Untrusted email input; must not leak user existence |
| /api/auth/reset/request -> Supabase | Service role key used for generateLink |
| /api/auth/reset/request -> Resend | Email delivery — must not be abused for spam |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-18-09 | Information Disclosure | /api/auth/reset/request | mitigate | Always return 200 regardless of email existence; never leak whether user account exists |
| T-18-10 | Denial of Service | /api/auth/reset/request | mitigate | 3/hr/email + 10/hr/IP rate limiting via authRateLimit |
| T-18-11 | Denial of Service | Password reset abuse | mitigate | Flag at 5/day/email with admin notification; advisory not auto-lock |
| T-18-12 | Spoofing | Resend email delivery | accept | Resend handles DKIM/SPF; domain already verified in existing setup |
| T-18-13 | Repudiation | Reset requests | mitigate | All requests logged to AuthEvent table with email, IP, timestamp |
</threat_model>
