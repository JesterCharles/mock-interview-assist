<objective>
Close the feedback loop: after Submit, poll the attempt endpoint with exponential backoff (500ms → 5s, stop at 60s wall) and surface queued/running/verdict states explicitly. Render the verdict card (overall pill + visible accordion + hidden pass/total), attach an attempt history sidebar scoped to the current challenge + caller, and handle network/rate-limit/forbidden errors via toasts.

Purpose: Without verdict surfacing, the solve page feels broken. This plan is the payoff — the moment the user sees their code judged.

Output: Complete, observable submission experience — queued → running → verdict. Attempt history visible. Monaco still lean. Zero hidden-test leakage.
</objective>

<!-- Phase 39 D-06 poll response (authoritative): -->

```typescript
interface AttemptPollResponse {
  attemptId: string;
  verdict: 'pass' | 'fail' | 'timeout' | 'mle' | 'runtime_error' | 'compile_error' | 'pending';
  score: number | null;              // 0-100 once resolved; null while pending
  visibleTestResults: Array<{
    caseId: string;
    passed: boolean;
    stdout: string;
    durationMs: number;
    // Note: stdin and expectedStdout are provided for visible tests (trainer-authored public info)
    stdin: string;
    expectedStdout: string;
  }>;
  hiddenTestResults: {
    passed: number;
    total: number;
    // NOTHING ELSE. No caseId, no stdin, no expectedStdout, no stdout.
  };
  submittedAt: string;
  completedAt: string | null;
}
```

Running/queued distinction: CONTEXT D-11 says states are `queued → running → verdict`. The Phase 39 D-06 response only models `verdict='pending'` for not-yet-resolved. Two options:
1. Extend Phase 39 response with a secondary `phase: 'queued' | 'running'` field — CLEANEST
2. Infer from elapsed time since `submittedAt`: < 1500ms → "queued", else → "running"

This plan does option 2 as a client-side inference (no server change needed). Document as a debt ticket for v1.5 to add explicit phase.

<!-- Attempt history endpoint (NEW in this plan): -->

```typescript
GET /api/coding/attempts?challengeId=<id>&limit=10
response 200: {
  items: Array<{
    attemptId: string;
    verdict: 'pass' | 'fail' | ... | 'pending';
    language: string;
    submittedAt: string;
    score: number | null;
  }>;
}
```

Scope: current caller only (server filters by associateId == caller.associateId OR caller is trainer viewing a specific associate — v1.4 only needs caller's own history per D-14).

The hook handles an already-resolved attempt gracefully (one-shot fetch returns terminal verdict, polling never starts).

  <behavior>
    - Calling `usePollAttempt(null)` → idle, no fetch
    - Calling `usePollAttempt(id)` → fires first fetch immediately
    - While response.verdict === 'pending': schedule next poll with backoff — delays [500, 750, 1125, 1687, 2531, 3796, 5000, 5000, ...] ms (factor 1.5, capped at 5000)
    - Total wall clock stops at 60000ms — hook stops polling, sets `status: 'timeout-waiting'`, surfaces a muted message "Taking longer than expected — check back soon"
    - When response.verdict is terminal (pass/fail/timeout/mle/runtime_error/compile_error): hook stops polling, sets `status: 'resolved'`
    - Changing `attemptId` resets the backoff clock + wall clock and cancels outstanding timer
    - Unmount cleans up timer (no stray fetches after unmount)
    - Inferred phase: if verdict==='pending' AND (now - submittedAt) < 1500 → `phase: 'queued'`; else `phase: 'running'` (documented in interfaces above)
    - Error handling: 403 → `error.code='FORBIDDEN'`; 404 → `error.code='NOT_FOUND'` + stop polling; 429 → respect Retry-After for next delay (clamp to remaining wall budget); network error → retry with backoff
    - Tests use `vitest.useFakeTimers()` to advance through the schedule
  </behavior>
  <action>

    - Constants: `INITIAL_DELAY = 500`, `MAX_DELAY = 5000`, `BACKOFF_FACTOR = 1.5`, `WALL_TIMEOUT_MS = 60000`
