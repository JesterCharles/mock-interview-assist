## Seed 1 â€” Wave-parallel planner as pipeline default

**Trigger:** Next milestone with â‰Ą 5 phases, OR any `/pipeline-plan` invocation.

**Context:** v1.4 planned 8 of 9 phases in ~26 minutes by dispatching planner subagents in parallel after Phase 36 shape locked. Previous milestones (v1.2, v1.3) ran plan stage serially and took hours. The parallelism did not hurt plan quality â€” all 8 plans produced coherent CONTEXT + PLAN artifacts consumed cleanly by execute stage.

**Action:** Make "plan all phases in parallel once the foundational phase shape locks" the pipeline default, not an emergent clever trick. Update `/pipeline-plan` to detect when phase 1 context is complete and dispatch the remaining plans in parallel waves keyed on explicit dependencies.

## Seed 2 â€” Real-Postgres SQL service (hardened isolated runner)

**Trigger:** Client-facing false-positive/false-negative on SQL challenges, OR trainer explicitly requests Postgres-flavored challenges, OR readiness signals from SQL feel low-confidence.

**Context:** v1.4 SQL ships SQLite-only via Judge0. Carryover from `v1.4-discovery-seeds.md` Seed 2, promoted here because after execution we now know SQL is the lowest-confidence signal channel in v1.4 â€” the result normalizer (`sqlResultNormalizer.ts`) had to tolerate 5-step pipeline slack that would not exist against real Postgres.

**Action:** Build dedicated hardened SQL service: prewarmed isolated schemas per attempt, role-locked no-superuser connection, `statement_timeout`, no extensions, no network egress, full teardown per attempt. Trainer reporting label graduates from `SQL fundamentals (SQLite dialect)` to separate `SQL (Postgres)` track.

## Seed 4 â€” Redis rate-limit migration

**Trigger:** MSA deploy goes live with > 1 app VM, OR Phase 43 `IAC-01` multi-instance rollout, OR observed rate-limit miss in production.

**Context:** `rateLimitService.ts` still uses `data/rate-limits.json` â€” per-pod filesystem state. Phase 43 deploys app + Judge0 on separate VMs but v1.4 stays single-app-instance. The moment NLM scales horizontally, the rate-limit file goes stale and abuse detection fails.

**Action:** Add Redis-backed rate-limit adapter behind the same `RateLimitStore` interface. Env switch: `RATE_LIMIT_STORE=file|redis`. Reuse Judge0's Redis instance if sized appropriately, or stand up a dedicated cache instance. Backfill integration tests against both adapters.

## Seed 6 â€” Dependency-aware phase executor ordering

**Trigger:** Every `/gsd-execute-phase` with â‰Ą 5 phases in a single autonomous run.

**Context:** Phase 42 (SQL MVP) initially executed before its prerequisites (37-41) because the autonomous executor processed phases in numeric order. P42 planner noted the blocker; P42 aborted and re-ran after 41 shipped. Cost: one abort + ~3h clock gap in the reflect timeline.

**Action:** Add explicit `Depends on` parsing to `gsd-analyze-dependencies` output + feed that DAG to `/gsd-execute-phase` as the traversal order. Fail loudly when numeric next-phase has unshipped dependencies instead of silently executing. Add a smoke test: autonomous pipeline run with a deliberately out-of-order dep-graph must pick the right order.

## Seed 7 â€” Autonomous-executor package.json write hook

**Trigger:** Any autonomous run where a subagent authors a file importing a package that isn't in `dependencies` or `devDependencies`.

**Context:** Phase 44 CR-01: `scripts/load-test-coding.ts` imported `p-limit` and used `tsx`; neither was in package.json. Agent wrote the script, tests referenced the script, but `npm run test:load` would have `Cannot find module 'p-limit'`. Codex-style review caught it; self-review wouldn't.

**Action:** Add a pre-commit hook (or subagent post-step) that greps new `import` statements against `package.json` and `node_modules/.package-lock.json`, failing with a prompt to run `npm install <pkg> --save-dev` before the commit lands. Alternatively, bundle this into `/gsd-code-review-fix` as a mandatory pre-check.

## Seed 8 â€” Live language-map query (eliminate JUDGE0_LANGUAGE_MAP drift)

**Trigger:** Judge0 version bump (patch or minor), OR new language added to challenge bank.

**Context:** JUDGE-06 spike surfaced drift: pre-spike map assumed Judge0 2.x IDs (`javascript: 93`, `typescript: 94`). Running 1.13.1 for CVE patch (GHSA-q7vg-26pg-v5hr) meant live IDs were `63` and `74`. Drift was silent because unit tests hard-coded the mock IDs. Every Judge0 version bump is a fresh drift risk.

**Action:** Replace hard-coded `JUDGE0_LANGUAGE_MAP` with a boot-time query to `/languages`, cached in-memory with 24h TTL + explicit refresh route. Map canonical slugs (`python`, `javascript`, ...) to discovered IDs by regex against language `name` field. Add a drift-detection test that fails CI if the cached map diverges from live IDs on a scheduled probe.

## Seed 9 â€” arm64 dev-host Judge0 bypass (QEMU namespace failure)

**Trigger:** New developer onboards on Apple Silicon or ARM64 Linux, OR dev-loop regression: "I can't run the coding feature locally."

**Context:** JUDGE-06 spike proved Judge0's `isolate` sandbox cannot run under QEMU userspace emulation on arm64 hosts. `clone(CLONE_NEWPID|CLONE_NEWUSER)` returns `EINVAL`. API/contract verification works; actual code execution does not. arm64 devs hit a dead end on sandbox-dependent work.

**Action:** Two paths:
- **Path A (dev convenience):** Ship a `judge0-mock` sidecar service that accepts `/submissions`, stores payloads, returns synthetic verdicts from a fixture table. Env switch: `JUDGE0_MOCK=1`. Good for UI dev, bad for execution-correctness work.
- **Path B (full stack):** Document arm64 dev workflow as: "run a x86_64 Linux VM locally (Lima / UTM)" or "SSH-tunnel to a shared dev GCE VM". Add `docs/dev-arm64-judge0.md`.

Recommend Path B for anyone touching Judge0 internals, Path A as an acceptable bypass for UI-only work.

## Seed 10 â€” Review-fix batching metric

**Trigger:** Any autonomous pipeline run that processes > 3 phases.

**Context:** v1.4 closed **49 review findings** across 3 review-fix batches (P36-38, P39-43, P42+P44). Fix latency was acceptable (minutes) but the peak batch of 13 fixes across 5 phases was high-variance â€” individual fixes blocked other fixes on file lock in worktrees. No alarm fired because we knew going in.

**Action:** Add `max_findings_per_batch` target (suggested: 10) to pipeline config. When a review batch exceeds it, dispatch multiple `/gsd-code-review-fix` subagents in parallel by phase. Emit telemetry: findings/hour, fixes/hour, fix-lag per finding. Surface as an autoresearch-adjacent metric if the batch regresses over time.
