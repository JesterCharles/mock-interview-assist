# Forward-Looking Seeds from v1.4 Discovery

Seeds planted during v1.4 discovery for future milestones. Each has a trigger condition that should surface it at the right time.

## Seed 1 — Function-level test harness (per-language drivers)

**Trigger:** Associates push back on stdin/stdout pedagogy, OR v1.5 planning when coding challenges exit beta.

**Context:** v1.4 ships stdin/stdout matching across all 5 languages. Codex confirmed this is sufficient for algorithms / control flow / hash maps / basic SQL but loses Python `pytest` mastery, JS/TS Jest signal, Java OO design, and proper .NET SDK fluency.

**Action:** Design per-language test drivers (LeetCode-style function wrapping). Ship as an opt-in challenge variant: challenge author declares `harness: "function"` in challenge metadata, provides function signature, driver injects user code into a language-specific runner. Start with Python + JS/TS (highest client demand), expand to Java + C#.

---

## Seed 2 — Real Postgres SQL runner (hardened isolated service)

**Trigger:** SQLite dialect creates client-facing false positives/negatives, OR trainer explicitly requests Postgres-flavored challenges.

**Context:** v1.4 SQL = SQLite-only via Judge0's built-in language. Sufficient for SQL fundamentals but diverges from real Postgres in window functions, CTEs with MATERIALIZED, JSON operators, array types, and many dialect-specific constructs.

**Action:** Build a separate hardened SQL execution service: prewarmed isolated schemas per attempt, role-locked connection (no superuser), `statement_timeout`, no extensions enabled, no network egress, full teardown per attempt. NOT a Judge0 extension — a dedicated Next.js + managed Postgres service. Trainer reporting label graduates from "SQL fundamentals (SQLite dialect)" to separate "SQL (Postgres)" track.

---

## Seed 3 — In-app challenge authoring (trainer editor)

**Trigger:** PR-based authoring becomes a bottleneck, OR trainers ask to edit challenges without GitHub.

**Context:** v1.4 ships PR-based authoring only — trainers edit markdown + starter code + test cases in GitHub. This keeps v1.4 scope bounded but assumes trainers are comfortable with git/PR workflow.

**Action:** Build an in-app challenge authoring UI: markdown editor (prompt), Monaco editor per language (starter), test-case editor (stdin/stdout pairs), "Run against hidden tests" preview, PR-auto-generation to the private hidden-tests repo on save. Keep repo as source of truth; UI is a convenience layer.

---

## Seed 4 — .NET 8+ modern runtime (separate sandbox)

**Trigger:** Client demand for modern ASP.NET, async/await, LINQ, or .NET 8+ specific features confirmed.

**Context:** v1.4 ships Judge0's C# Mono language (legacy C# console support). Insufficient for modern .NET fluency. Codex flagged as a week-6 risk.

**Action:** Stand up a dedicated .NET 8+ sandbox (Docker container with the .NET SDK, restricted network, cgroups CPU/mem limits). Add as a new Judge0-adjacent language path: `JUDGE0_URL` handles Python/JS/TS/Java/SQL/Mono; new `DOTNET_RUNNER_URL` handles modern .NET. Env-driven so it's optional infra.

---

## Seed 5 — Multi-modal practice platform (design + system design + behavioral)

**Trigger:** v2.0 vision planning, OR client demand validates coding challenges as a category extension worth doubling on.

**Context:** v1.4 establishes coding challenges as a second rep type alongside mock interviews. Long-term positioning as a multi-modal practice platform (interview + code + design + system design + behavioral simulations) is a moat — continuous skill telemetry across rep types, not snapshot audits.

**Action:** Planning doc for v2.0 multi-modal expansion. Identify shared primitives (GapScore input abstraction, attempt lifecycle, readiness signal aggregation). Prototype design-challenge format (system design diagram submission + AI review) and behavioral-simulation format (trainer-led or AI-led scenario scoring).

---

## Seed 6 — Anti-cheat heuristics (AI-generated detection, sample memorization)

**Trigger:** Client trust signals require it, OR associates visibly hardcode sample answers at scale.

**Context:** v1.4 relies on hidden tests to defeat sample-memorization but does NOT detect AI-generated submissions (Copilot, ChatGPT, Cursor). Codex flagged as a real risk once the feature ships.

**Action:** Add heuristics layer: time-to-first-submit tracking (AI-generated submissions tend to be fast and correct on first try), edit-burstiness analysis (paste of large blocks is a tell), typing-pattern fingerprint (optional, invasive), style-profile diff against past submissions. Start with time-to-first-submit + paste detection; avoid keystroke logging.
