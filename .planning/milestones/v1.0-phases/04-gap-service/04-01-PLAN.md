**Step 1: Add GapScore model to prisma/schema.prisma (per D-06)**

Add the GapScore model after existing models:
```prisma
model GapScore {
  id            String    @id @default(cuid())
  associateId   String
  skill         String
  topic         String    @default("")
  weightedScore Float
  sessionCount  Int
  lastUpdated   DateTime  @updatedAt

  associate     Associate @relation(fields: [associateId], references: [id], onDelete: Cascade)

  @@unique([associateId, skill, topic])
  @@index([associateId])
}
```

IMPORTANT per research Pitfall 1: `topic` is `String` NOT `String?`. Use `@default("")` -- empty string is the sentinel for skill-level scores. NULL in PostgreSQL composite unique constraints creates duplicates. This is non-negotiable.

Add the reverse relation to the Associate model:
```prisma
// Inside model Associate, add:
gapScores   GapScore[]
```

**Step 2: Add techMap field to Session model**

Add to the existing Session model (after assessments or similar Json field):


This resolves the skill identity gap (research Pitfall 2). weekNumber is an integer key; the value is the skill name derived from GitHubFile.name (e.g., "react.md" -> "react").
