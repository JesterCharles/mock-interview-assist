**Step 1: Create `src/lib/gapPersistence.ts`**

This module bridges the pure algorithm and the database. Two exported functions:

**saveGapScores(associateId: string): Promise&lt;void&gt;**
1. Query all completed sessions for this associate from Prisma, ordered by createdAt DESC (newest first)
2. Convert Prisma Session records to InterviewSession-compatible objects (assessments is Json, techMap is Json -- parse them)
3. Call `computeGapScores(sessions)` from gapService.ts
4. For each GapScoreInput, upsert into GapScore table:
```typescript
await prisma.gapScore.upsert({
  where: {
    associateId_skill_topic: { associateId, skill: input.skill, topic: input.topic }
  },
  update: {
    weightedScore: input.weightedScore,
    sessionCount: input.sessionCount,
  },
  create: {
    associateId,
    skill: input.skill,
    topic: input.topic,
    weightedScore: input.weightedScore,
    sessionCount: input.sessionCount,
  },
});
```
5. Delete any GapScore records for skills/topics that no longer appear in computed results (stale cleanup)

**getGapScores(associateId: string): Promise&lt;GapScoreResult&gt;**
1. Count completed sessions for this associate
2. If count < 3, return `{ gated: true, sessionCount, requiredSessions: 3, scores: [] }` (per D-05, GAP-03)
3. If count >= 3, query all GapScore records for this associate, return `{ gated: false, sessionCount, scores: [...] }`

Return type:
```typescript
export interface GapScoreResult {
  gated: boolean;
  sessionCount: number;
  requiredSessions?: number;  // only present when gated
  scores: Array<{
    skill: string;
    topic: string;
    weightedScore: number;
    sessionCount: number;
    lastUpdated: Date;
  }>;
}
```

**Step 2: Wire gap computation into /api/history POST**

In `src/app/api/history/route.ts`, after the existing dual-write (file write + Supabase write from Phase 2), add fire-and-forget gap computation:

```typescript
import { saveGapScores } from '@/lib/gapPersistence';

// After successful session save (both file and DB):
// Only compute gaps if session has an associate linked
if (dbSession?.associateId) {
  saveGapScores(dbSession.associateId).catch((err) => {
    console.error('[gap-service] Failed to update gap scores:', err);
```

CRITICAL per D-04 and research: This is fire-and-forget. Do NOT await this call in the response path. Do NOT let gap computation failure cause a 500 response. The session save response must return immediately after the dual-write succeeds.

The exact variable name for the DB session result depends on Phase 2's implementation -- read the current route.ts to find it. Look for the Prisma create/upsert result that has `associateId`.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-05 | Information Disclosure | /api/associate/[slug]/gaps | mitigate | Auth guard with isAuthenticatedSession() -- unauthenticated requests get 401, not 404, preventing slug enumeration |
| T-04-06 | Spoofing | /api/associate/[slug]/gaps | mitigate | Slug validated with zod regex (alphanumeric + hyphens only), preventing injection |
| T-04-07 | Tampering | gapPersistence.ts | mitigate | Gap scores computed from DB data (not user input). No external input to gap computation beyond the associateId derived from authenticated session save. |
| T-04-08 | Denial of Service | saveGapScores | accept | Fire-and-forget pattern means gap computation cannot block session saves. Computation bounded by session count per associate (< 100 sessions due to cap). |
| T-04-09 | Elevation of Privilege | /api/associate/[slug]/gaps | mitigate | Single-password auth means any authenticated user can view any associate's gaps. Acceptable for MVP (single training org). Multi-tenancy is v2 (SCALE-01). |
