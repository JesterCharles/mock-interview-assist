<objective>
Ship the PIN generation and verification backbone: schema fields, pure PIN service, and three API endpoints (generate, verify, logout). No middleware changes, no UI. This plan delivers AUTH-01 (per D-01..D-05) and the server half of AUTH-02 (per D-14..D-17), with the revocation-capable cookie design from D-08/D-09a/D-09b (Codex finding #4).

Purpose: Establish the cryptographic and persistence foundation for associate auth before wiring middleware or UI. Isolating this plan lets the other two plans build in parallel against a stable contract. A dedicated `ASSOCIATE_SESSION_SECRET` decouples associate auth from the trainer password; embedding `pinGeneratedAt` as the token version gives us a DB-backed revocation path with zero schema additions beyond `pinHash`/`pinGeneratedAt`.

Output: Schema migration, `pinService.ts`, `associateSession.ts` (dedicated-secret + version-aware), three route handlers, unit tests for pinService + token signing, `.env.example` updated.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| anonymous client → /api/associate/pin/verify | Untrusted PIN + slug cross into bcrypt comparison and cookie issuance |
| anonymous client → /api/associate/pin/generate | Must be blocked by trainer-auth guard |
| cookie store → server | associate_session cookie value parsed + HMAC-verified + version-checked before trust |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-01 | Spoofing | /api/associate/pin/verify | mitigate | HMAC-signed token using dedicated ASSOCIATE_SESSION_SECRET (D-08, D-09b); tampered tokens fail timing-safe signature check |
| T-09-02 | Tampering | associate_session cookie | mitigate | Opaque signed token; any payload mutation invalidates signature; HttpOnly prevents client JS tampering (D-07) |
| T-09-03 | Information Disclosure | PIN in logs/responses | mitigate | Plaintext PIN returned exactly once on generate; never logged; verify never echoes PIN (D-04, D-25) |
| T-09-04 | Denial of Service | PIN brute force | mitigate | bcrypt cost 10 + fingerprint rate limit 5/15min (D-02, D-17) |
| T-09-05 | Elevation of Privilege | /api/associate/pin/generate | mitigate | Hard trainer-auth guard via isAuthenticatedSession(); 401 otherwise (D-14) |
| T-09-06 | Information Disclosure | Slug enumeration via verify | accept | Rate limiter + identical 401 response for wrong-slug and wrong-pin reduces oracle; slugs already non-secret in v1.0 (D-23) |
| T-09-07 | Tampering | Modulo bias in PIN generation | mitigate | Use crypto.randomInt(0, 1_000_000) — uniform — not Math.random or modulo (D-01) |
| T-09-17 | Elevation of Privilege | Trainer password rotation invalidating associate sessions | mitigate | Dedicated ASSOCIATE_SESSION_SECRET decoupled from APP_PASSWORD (D-09b, Codex #4) |
| T-09-18 | Elevation of Privilege | Compromised PIN cookie remaining valid after regeneration | mitigate | Token version = pinGeneratedAt; regeneration advances version; consumers compare (D-09a, Codex #4) |
</threat_model>
