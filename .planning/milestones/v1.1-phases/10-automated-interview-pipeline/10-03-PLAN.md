<objective>
Deliver the DB-backed readiness recompute repair path required by Codex finding #5. An operator (trainer or external cron) can POST `/api/admin/readiness-sweep` to reprocess any Sessions whose `readinessRecomputeStatus` is `pending` or `failed`. The sweep groups affected sessions by associate, invokes `runReadinessPipeline` once per associate, and transitions session markers accordingly.

Purpose: Ensures that if a deploy/restart/DB hiccup drops a readiness recompute mid-flight, the system has a concrete, observable, rerunnable path to correctness — not a silent stale state.

Output: `src/lib/readinessSweep.ts` (pure logic) + `src/app/api/admin/readiness-sweep/route.ts` (trainer-auth wrapper) + tests.
</objective>

    - Test 5: Pipeline throws for one associate → that associate's sessions remain 'failed', sweep continues for other associates
    - Test 6: After successful run, 'pending' Sessions transition to 'done' (verified via runReadinessPipeline's own status updates)
    - Test 7: Idempotent — running sweep twice with no new failures yields SweepResult with 0 examined on the second run
  </behavior>
  <action>
    Create `src/lib/readinessSweep.ts` exporting `runReadinessSweep(opts?: { batchSize?: number })`.

    Logic:
    1. Default `batchSize = 50`.
    2. Query affected associates:
       ```ts
       const rows = await prisma.session.findMany({
         where: {
           readinessRecomputeStatus: { in: ['pending', 'failed'] },
           associateId: { not: null },
         },
         select: { id: true, associateId: true },
         orderBy: { createdAt: 'asc' },
       });
       ```
    3. Group session ids by associateId. Take at most `batchSize` DISTINCT associates.
    4. For each associate in the batch: pick the most recent session id as the "marker" session, call `runReadinessPipeline(associateId, sessionId)`. The helper updates that session's status; ALSO update any other pending/failed sessions for this associate to `done` in a single `updateMany` call after successful recompute (since one recompute covers all the associate's outstanding markers).
    5. On error from pipeline: leave the associate's sessions as `failed`, increment failureCount, continue.
    6. Return SweepResult.

    Wrap each associate's iteration in try/catch so one bad actor doesn't abort the whole sweep.

    Log prefix `[readiness-sweep]`.

    Write Vitest tests mocking `@/lib/prisma` and `@/lib/readinessPipeline`. Cover all 7 behaviors.
  </action>
